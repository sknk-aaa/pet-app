# 06. 管理画面仕様

このドキュメントは Next.js で構築する管理画面の機能と画面構成を定義します。

---

## 目的と利用者

管理画面は、運営者(管理者)が「今日のペット」の運用と不適切コンテンツ対応を行うための Web 管理ツールです。

利用者:
- アプリの運営者(初期は開発者本人 1 名)
- 将来的に追加される運営スタッフ

---

## 技術スタック

- フレームワーク: Next.js (App Router)
- UI コンポーネント: Tailwind CSS + shadcn/ui
- データアクセス: Supabase JS Client(anon key 使用、ログインユーザーの is_admin を RLS で判定)
- ホスティング: Vercel
- 認証: Supabase Auth(モバイルアプリと同じユーザーテーブル)

詳細は `docs/07-tech-stack.md` 参照。

---

## アクセス制御

- 管理画面は Supabase Auth でログインが必須
- `users.is_admin = true` のユーザーのみがアクセス可能
- ログイン後、is_admin 判定で「アクセス権限がありません」を表示する場合あり
- 初期管理者は SQL で手動付与:
  ```sql
  UPDATE users SET is_admin = true WHERE id = '<管理者ユーザーの uuid>';
  ```
- service_role キーは使わない(認証されたユーザーセッションを通じてのみアクセス)

## 1. ログイン画面

- メールアドレス + パスワードでログイン(管理者用に Email 認証メインとする)
- Apple / Google ログインも併設してよい
- ログイン後、is_admin チェックを実施
- false の場合「アクセス権限がありません」を表示しログアウト
- 自動ログイン(セッション永続化)あり

- 「抽選推薦待ち」: `SELECT COUNT(*) FROM featured_candidates WHERE status = 'scheduled' AND entry_date = 昨日(JST)`
- 「pending」: 直近 24 時間以内の pending 数
- 「ユーザー総数」: `SELECT COUNT(*) FROM users`

## 3. 今日のペットレビュー

最も使用頻度が高い画面。

### 画面構成

```
┌──────────────────────────────────────────────────────────┐
│ 今日のペットレビュー                                     │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  対象日: 2026年5月27日(明日掲載)                       │
│                                                          │
│  ── 推薦候補 ──────────────────────────────────────── │
│  ┌──────────────────────────────────────────────────┐  │
│  │ [大きな写真]                                      │  │
│  │ タイトル: 窓辺で寝るポチ                          │  │
│  │ ペット: ポチ(ねこ)                              │  │
│  │ 投稿者: user@example.com (登録 2026/3/12)        │  │
│  │ 連続記録: 12 日                                   │  │
│  │ 過去の掲載: 0 回                                  │  │
│  │ 通報件数: 0                                       │  │
│  │                                                  │  │
│  │ [✓ 承認] [✗ 却下] [別の候補から選ぶ ▾]          │  │
│  └──────────────────────────────────────────────────┘  │
│                                                          │
│  ── その他の pending 候補 ───────────────────────── │
│  グリッド表示(写真サムネイル + 簡易情報)              │
│  クリックでこの候補を「推薦」と入れ替え                  │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 動作

- 「✓ 承認」: 推薦候補の status を `scheduled` → `approved` に変更
- 「✗ 却下」: 推薦候補の status を `rejected` に変更
- 「別の候補から選ぶ」: 別の pending 候補一覧から 1 枚選び、入れ替える
  - 元の scheduled は `pending` に戻すか、`rejected` にする(管理者の判断)
  - 入れ替え先を `scheduled` に変更
- 既に approved になっている場合は「✓ 承認済み」表示

## 4. 投稿候補一覧

すべての featured_candidates を一覧できる画面。

### フィルター

- 投稿日(範囲指定)
- ステータス(複数選択可: pending / scheduled / approved / featured / rejected / withdrawn / hidden)
- 投稿者(メールアドレス検索)
- 通報件数あり / なし

### 一覧表示

テーブル形式:

| サムネイル | 投稿日 | 投稿者 | タイトル | ペット | ステータス | 通報数 | アクション |
|---|---|---|---|---|---|---|---|

- サムネイル列はクリックで詳細モーダル(大きな写真)
- ステータスはバッジ表示で色分け(pending = 灰、scheduled = 青、approved = 緑、featured = 金、rejected = 赤、withdrawn = グレー、hidden = 黒)

### アクション

候補ごとの操作:

- 詳細表示
- 承認(pending → approved)
- 却下(任意の状態 → rejected)
- 非表示化(featured → hidden)

ステータスごとに可能なアクションは制限する(例: featured を pending に戻すことはできない)。

## 6. 通報一覧

reports の一覧。

### 一覧表示

| 通報日 | 通報者 | 対象写真サムネ | 理由 | 詳細 | ステータス | アクション |
|---|---|---|---|---|---|---|

### フィルター

- ステータス(open / reviewed / resolved)
- 理由
- 期間

### アクション

- 「対応済みにする」: status を `reviewed` に
- 「写真を非表示にする」: 対象の featured_pets の status を `hidden` に、reports の status を `resolved` に
- 「ユーザーを BAN にする」: 投稿者 user の `banned_at` を now() に、reports の status を `resolved` に
- 「却下(通報を無視)」: reports の status を `resolved` に

## 7. ユーザー管理

### 一覧表示

| email | 登録日 | 投稿数 | 通報数(された) | 通報数(した) | is_admin | 状態 | アクション |
|---|---|---|---|---|---|---|---|

「状態」は normal / banned / admin の表示。

### アクション

- BAN にする: `banned_at = now()` を設定
- BAN を解除: `banned_at = NULL`
- 管理者権限付与: `is_admin = true`
- 管理者権限剥奪: `is_admin = false`

自分自身の `is_admin` を剥奪することはできない(うっかり防止)。

### BAN の効果

`users.banned_at IS NOT NULL` のユーザーは:

- 今日のペットへの投稿不可(クライアント側でチェック)
- 既存の featured_pets / featured_candidates は残るが、新規投稿時に Edge Function で拒否
- リアクション・通報も不可

具体的な実装は RLS ポリシーで:

```sql
-- featured_candidates の INSERT ポリシーを更新
CREATE POLICY "non-banned can insert candidate"
  ON featured_candidates FOR INSERT
  WITH CHECK (
    auth.uid() = user_id
    AND status = 'pending'
    AND NOT EXISTS (
      SELECT 1 FROM users WHERE id = auth.uid() AND banned_at IS NOT NULL
    )
  );
```

## デプロイと運用

### デプロイ

- Vercel に GitHub 連携でデプロイ
- 本番環境変数: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- カスタムドメインの設定は任意

### 運用フロー(管理者の日常タスク)

朝(07:00 までに):
1. 「今日のペットレビュー」を開く
2. 抽選で推薦された候補を確認
3. 適切なら承認、問題があれば差し替え or 却下

随時:
4. 通報一覧に新着があれば対応
5. ダッシュボードで状況確認

新規管理者追加:
6. ユーザー管理から該当ユーザーの is_admin を true に設定

---

## 管理画面の非機能要件

- レスポンシブ対応は最低限(PC で使えればよい、モバイル最適化は不要)
- リアルタイム更新は不要(ページリロードで最新を取得)
- 操作ログ機能は v1.0 では不要
- データエクスポート機能は v1.0 では不要(必要なら Supabase のダッシュボードから SQL 実行)
