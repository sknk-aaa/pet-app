# 09. 例外処理・空状態・オフライン挙動・テスト原則

このドキュメントはエッジケースの仕様とテスト方針をまとめます。

---

## 空状態の表示

各画面で記録ゼロ・データなしの状態が発生したときの挙動を定義します。

| 画面 | 空状態の条件 | 表示内容 |
|---|---|---|
| ホーム(未記録時) | 思い出表示できる過去記録なし | 「これから始まる毎日を残しましょう」 |
| ホーム(未記録時 + ペットフィルター) | 選択中ペットの記録なし | 「(ペット名)の記録はまだありません」 |
| カレンダー | 記録ゼロ | グリッドは表示する。空セルのみ |
| カレンダー(ペットフィルター時) | 選択中ペットの記録なし | グリッドは表示する。空セルのみ |
| 記念日一覧 | 記念日タグ付き記録なし | 「特別な日に記念日タグを付けて、思い出を残しましょう」 |
| 今日のペット | 候補がない日 | 「本日のペットは準備中です」「明日の掲載候補を募集中です」 |
| 掲載履歴 | 自分の featured 記録なし | 「まだ掲載された写真がありません」 |
| ペット一覧 | 0 件(理論上発生しないが) | オンボーディングへ強制リダイレクト |
| 通報フォーム | - | エラー画面ではないので空状態なし |

空状態には以下のいずれかを含めること:

- 状況を説明する短い文(否定的でない、優しい表現)
- 次にできるアクション(あれば)
- 軽いイラストまたはアイコン(任意、過度な装飾は避ける)

## オフライン時の挙動

各機能のオフライン時の挙動を定義します。

### オフラインで完全に動く

これらはネット接続なしでも完全に機能する:

- 写真記録(撮影・タイトル・メモ・記念日タグ・写っているペット選択)
- ローカル保存(SQLite + FileSystem)
- カメラロール保存(設定が ON の場合)
- カレンダー閲覧
- ホーム閲覧
- 記念日一覧
- ペット設定の閲覧・編集
- 連続記録日数の更新
- 通知リマインダーのスケジュール
- Pro 機能の利用(月まとめ生成、ペット別表示など。Pro 状態が grace period 内の場合)

### オフラインで操作だけ可、送信は後で再試行

- 今日のペット参加 ON で保存 → ローカル保存は完了、送信は pending_uploads にキュー
- リアクション追加 → 楽観的更新、送信キュー
- リアクション取り消し → 楽観的更新、送信キュー
- 通報送信 → 送信キュー(後でログイン状態確認の上送信)
- 今日のペット候補の取り下げ → 送信キュー

操作完了の UI フィードバックは即座に出す。サーバー反映は遅れる旨を必要に応じて軽く表示。

### オフラインで動かない

- ログイン処理(Apple / Google / メール) → 「ネットワーク接続を確認してください」
- 今日のペット閲覧(キャッシュなしの場合) → 「ネットワーク接続を確認してください、再試行する」
- 掲載履歴の取得 → キャッシュがあれば表示、なければ再試行
- Pro 購入 / 復元 → 「ネットワーク接続を確認してください」
- アカウント削除 → ネット復帰を待つ

### オフライン検出

- `@react-native-community/netinfo` で接続状態を監視
- 接続断時に画面上部に細いバナーで「オフライン中です」と表示(任意)
- 接続復帰時に pending_uploads を自動フラッシュ

## エラー処理

### 操作ごとのエラー対応

| 操作 | 主なエラー | UI 対応 |
|---|---|---|
| 写真撮影 | カメラ権限拒否 | 「カメラへのアクセスを許可してください。設定アプリで変更できます」+ 設定アプリへのリンク |
| アルバム選択 | フォトライブラリ権限拒否 | 同上 |
| 写真保存(リサイズ失敗) | 元画像が壊れている、メモリ不足 | 「写真の保存に失敗しました。もう一度お試しください」 |
| 写真保存(ディスクフル) | 容量不足 | 「端末の空き容量が不足しています。空き容量を確保してから再試行してください」 |
| 候補送信(ネット) | 接続なし、サーバーエラー | 楽観的にローカル保存完了、バックグラウンドで再試行、ユーザーには「送信失敗、タップで再送」表示 |
| リアクション送信 | サーバーエラー | 楽観的更新は維持、裏でリトライ。3 回失敗で諦める(UI には影響しない) |
| ログイン失敗 | 認証拒否 | 「ログインに失敗しました。再度お試しください」 |
| Pro 購入失敗 | ユーザーキャンセル / 決済失敗 | 「購入を完了できませんでした」or 何もしない(ユーザーキャンセル) |
| Pro 復元失敗 | 復元可能な購入なし | 「復元できる購入が見つかりませんでした」 |
| 通報送信失敗 | サーバーエラー | 「通報の送信に失敗しました。後ほど再試行してください」 |

### グローバルエラーハンドリング

- React Error Boundary でアプリ全体のクラッシュを捕捉
- 致命的エラー時は「予期せぬエラーが発生しました。アプリを再起動してください」を表示
- 開発時のみ Sentry や Expo のエラーレポートに送信(本番ではユーザー同意のもとで)

## エッジケース一覧

### 写真記録まわり

**1 日 1 回しか保存できないが、当日に複数回フォームを開いた場合**
- 既に当日の entries が存在する → フォームは編集モードで開く
- 「新規作成」と「編集」の見分けは entries テーブルに今日の date があるかどうか

**写真撮影をキャンセル**
- フォームは表示せずホーム画面に戻る

**フォーム入力中にアプリを閉じた / バックグラウンドに**
- フォームの一時保存はしない(下書き機能はない)
- ユーザーが意図的にキャンセルしたと見なす
- 写真の一時ファイルは tmp/ に残るが、起動時の整合性チェックで削除

**端末時計が大きくずれている**
- 端末時計を信用する(v1.0 ではサーバー時刻と照合しない)
- 結果として連続日数が不正確になる可能性はあるが、許容

### 今日のペット参加まわり

**ログインしていない状態で参加トグルを ON にしようとした**
- ログイン画面に誘導
- ログイン成功後、フォームに戻ってトグルを ON 状態にする(ユーザーが意図したアクションを尊重)

**送信中にネット断**
- pending_uploads に残し、バックグラウンドで再試行
- 翌日 0:00 を過ぎたら自動削除

**サーバー側の候補がすでに削除されていた状態でクライアントから取り下げを試みた**
- RPC エラーになる
- クライアント側で「すでに処理済みです」と表示し、ローカルキャッシュをクリア

**承認後にユーザーが取り下げ → 管理者が「あれ?消えてる」**
- 管理画面では候補一覧から withdrawn ステータスで確認できる
- 一度 withdrawn になったものは復活させない

### リアクションまわり

**未ログインユーザーがリアクションを押したあとログインした**
- device_id でつけたリアクションはそのまま残る
- 同じ写真に対してログイン後にもリアクションすると、user_id でも別レコードが作成される(重複扱いだが許容)
- ログイン後の振る舞いを統一したい場合、ログイン時に device_id ベースのリアクションを user_id に紐づける処理を入れることもできるが、v1.0 では実装しない

**同じ写真への通報を 2 回しようとした**
- UNIQUE 制約でエラー
- クライアント側で「すでに通報済みです」を表示してフォームを閉じる

### Pro 状態まわり

**月額プランの自動更新が失敗 → expires_at を超過**
- 起動時の検証で気付く
- Pro 機能を無効化、警告メッセージ:
  「Pro が更新されていません。設定の Apple ID から購読を確認してください」

**買い切り後に間違って月額を購入しようとした**
- 既に有効な Pro 状態であることを検出し、「すでに Pro が有効です」を表示
- 購入を続行させない

**機種変更してアプリを再インストール**
- ローカルデータはすべて消える(クラウドバックアップは v1.0 ではない)
- Pro 状態は「購入を復元する」ボタンで復元可能
- ログインすればサーバーデータ(candidates、reactions など)にはアクセスできるが、ローカルの写真記録は復元できない

### ペットまわり

**写っているペットを 1 匹も選ばずに保存(Pro 複数登録時)**
- バリデーションエラー: 「写っているペットを 1 匹以上選んでください」

**ペット削除時、選択中ペットが消える**
- 別のペットに自動切り替え
- ペットが 0 になる削除は禁止(最後の 1 ペットは削除不可)

**Pro 解約後に複数ペットが残っている**
- 月額解約や買い切り無効化(本来発生しない)で Pro 機能が失われた場合
- ローカルデータは消さない(複数 ペット情報は残る)
- 写真記録時に「写っているペット」の表示が消え、選択中ペットが自動的に紐づく
- ペット選択 UI を見れば全ペットの存在は確認できる(機能制限ではなく見えるだけ)

### 今日のペット閲覧まわり

**画面オープン時に通信エラー**
- キャッシュがあれば表示
- なければ「ネットワーク接続を確認してください、再試行する」

**今日の掲載が hidden 化された**
- view から外れる(visible のみが見える)
- 画面リロード後は「準備中」表示

**自分が掲載されたが、まだプッシュ通知を受け取っていない**
- アプリ起動時に featured_pets を user_id でクエリして自分のものがあれば「今日掲載されました!」のダイアログ
- streak_state の featured_weight_streak リセット処理を実行

### 通知まわり

**通知許可をオンにせずアプリを使い続けた**
- リマインダー通知は機能しない
- 設定画面に「通知を有効にすると毎日のリマインダーが届きます」と案内
- 起動 X 回ごとに再度許可を求めるダイアログ(任意、過剰にならない範囲で)

## テストの原則

このアプリでどこをテストすべきか、原則だけ示します。具体的なテストコードの記述は Claude Code に委ねます。

### テストの優先度

優先度高(必ずテスト):

1. **連続記録日数のロジック** — `display_streak` と `featured_weight_streak` の更新、リセット条件。ロジックの中核なのでバグると体験が崩れる
2. **タイムゾーン処理** — Asia/Tokyo 固定の日付判定。「今日」「昨日」の境界判定
3. **抽選ロジック** — 重み付き抽選が偏らないか、min/max が正しく適用されるか
4. **取り下げの状態遷移** — pending / approved / scheduled / featured で挙動が変わる
5. **写真の前処理** — リサイズ後のサイズ、quality、サムネイル生成
6. **送信キューの動作** — リトライ回数、指数バックオフ、翌日経過後の削除

優先度中(できるだけテスト):

7. **RLS ポリシー** — 他人のデータを取得 / 書き込みできないこと
8. **未ログインリアクション** — device_id ベースで正しく動くか
9. **Pro 状態の検証** — 期限切れ判定、grace period
10. **ペット選択シートの遷移** — 無料 → 課金画面、Pro → ペット追加

優先度低(必要に応じて):

11. **UI スナップショット** — UI コンポーネントの見た目チェック
12. **管理画面の操作** — E2E テスト

### テストの種類

| 種類 | 対象 | ツール |
|---|---|---|
| ユニットテスト | ロジック関数、ヘルパー、状態管理 | Jest / Vitest |
| 統合テスト | DB アクセス、Supabase クライアント | Jest + Supabase ローカル環境 |
| E2E テスト | ユーザー操作フロー | v1.0 では必須ではない。手動テスト中心 |

### テスト用フィクスチャ

以下のフィクスチャを用意すると効率的:

- 連続日数の各パターン(0 日 / 1 日 / 14 日 / 30 日 / 100 日)
- entry の各種パターン(タグあり / なし / 写ってるペット 1 匹 / 複数)
- 今日のペット候補の各ステータス
- Pro 状態(未購入 / 買い切り / 月額有効 / 月額期限切れ)

### 手動テスト

App Store 提出前に必ず確認:

- iPhone 実機(複数モデルあれば望ましい)
- 通知許可 / 拒否 両パターン
- カメラロール許可 / 拒否
- オフライン状態でのアプリ動作
- 機種変更からのデータ移行(再インストール時の挙動)
- 課金フロー(Sandbox 環境)
- 課金復元
- アカウント削除
- ログアウト / 再ログイン

### CI

- GitHub Actions で push 時にユニットテスト・型チェック・lint を実行
- メインブランチへのマージ前に全テストパス必須
- iOS ビルドは EAS Build に委譲

## 監視とログ

v1.0 では最小限とする:

- Expo の標準クラッシュレポート
- Supabase のログ(Database / Auth / Functions)
- App Store Connect の標準分析

Sentry など外部監視サービスを v1.x で追加検討。
